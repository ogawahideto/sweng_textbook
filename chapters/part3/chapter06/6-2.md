# 6.2 絶え間なき循環——CI/CDパイプライン

**執筆状態**: 🟨 AI草稿

---

## 導入: ソフトウェアの血液循環

生きている動物の体内では、心臓が絶えず血液を送り出し、酸素と栄養を全身に届けています。ソフトウェアも同じです。
新しいコード（酸素）が書かれたら、それをテストし、ビルドし、ユーザーの手元（全身）へ届ける。この循環が止まると、ソフトウェアは鮮度を失い、やがて腐敗してしまいます。

この循環を自動化し、絶えず回し続ける仕組み。それが**CI/CDパイプライン**です。
これは単なる自動化ツールではありません。開発チームのリズムを作り、ソフトウェアに「命」を吹き込み続けるための心臓部なのです。

---

## CI: 継続的インテグレーション (Continuous Integration)

「インテグレーション（統合）」とは、あなたが書いたコードを、チームのメインブランチに合流させることです。

かつては、開発者が何週間も手元でコードを温め、リリース直前に全員で「統合作業」を行っていました。これは「統合地獄（Integration Hell）」と呼ばれ、阿鼻叫喚の修羅場でした。

CIの哲学はシンプルです。**「毎日、毎時間、少しずつ統合せよ」**。
コードをPushするたびに、クラウド上で自動的にテストが走り、「今の変更は安全か？ 他の機能を壊していないか？」を即座に判定します。これにより、バグは生まれた瞬間に発見され、修正されます。

---

## CD: 継続的デリバリー (Continuous Delivery)

CIでテストをパスした「清潔なコード」を、いつでもリリースできる状態にしておくこと。さらには、本番環境への展開まで自動化してしまうこと。それがCDです。

- **Continuous Delivery**: 本番へのデプロイは人間がボタンを押すが、そこまでの準備は自動化されている。
- **Continuous Deployment**: テストが通れば、人間の承認なしで自動的に本番へ反映される。

どちらを選ぶかはプロジェクトによりますが、目指すのは「リリースのコストを限りなくゼロにする」ことです。リリースが簡単になれば、私たちは「月に1回の大きなリリース」ではなく、「1日に10回の小さな改善」を選べるようになります。

---

## パイプラインの構築: GitHub Actions

現代のアルケミストにとって、最も身近なパイプライン構築ツールは **GitHub Actions** でしょう。
`.github/workflows` フォルダにYAMLファイル（呪文の書）を置くだけで、GitHubがあなたの執事（Runner）となり、指示通りに動いてくれます。

### ワークフローの例

```yaml
name: QuestForge CI/CD

on: [push]  # 呪文のトリガー: コードがPushされたら

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run Tests  # 1. テストを実行
        run: python -m unittest discover tests

  deploy:
    needs: test  # テストが成功した場合のみ実行
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Production  # 2. 本番へデプロイ
        run: ./deploy_script.sh
```

このように、「テスト → ビルド → デプロイ」という流れを定義しておけば、あなたはコードを書くことだけに集中できます。あとはパイプラインが、あなたのコードを安全に世界まで運んでくれるのです。

---

## まとめ

1.  **循環させる**: コードは鮮度が命。溜め込まず、常に統合し、循環させる。
2.  **CI (統合)**: 自動テストというフィルターを通し、常に「壊れていない状態」を保つ。
3.  **CD (デリバリー)**: リリースの手間を極小化し、ユーザーへの価値提供を加速させる。
4.  **パイプライン**: GitHub Actionsなどで自動化された手順を構築し、人間を単純作業から解放する。

CI/CDパイプラインが整ったプロジェクトでは、開発者は安心してコードを書き、失敗を恐れずに挑戦できます。それはまるで、強力なバックアップチームがついているF1レーサーのようなものです。

## AIへの詠唱例

この節で学んだことを実践するためのプロンプト：

```
GitHub Actionsのワークフローファイルを書きたいです。
以下の要件を満たすYAMLファイルのひな形を作成してください。
1. mainブランチへのPushをトリガーにする
2. Python 3.11をセットアップする
3. 依存ライブラリ（requirements.txt）をインストールする
4. Pytestでテストを実行する
5. テストが通ったら、Slackに通知を送る（通知ステップはダミーでOK）
```

```
CI/CDにおける「継続的デリバリー」と「継続的デプロイメント」の違いを、
非エンジニアのステークホルダーにも分かるように、
「レストランの厨房と配膳」に例えて説明してください。
```
