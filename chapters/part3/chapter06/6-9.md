# 6.9 賢者の工房——プラットフォーム・エンジニアリングとDevEx

6.1節でデプロイメント戦略を、6.2節でCI/CDパイプラインを設計しました。これらは「どう届けるか」の仕組みです。しかし開発チームが5人から50人、さらに500人へとスケールするとき、別の問いが生まれます。「毎日のように新しいマイクロサービスが立ち上がるこの組織で、誰もが同じ品質・速度でデプロイできるにはどうすればいいか？」

その答えが**プラットフォーム・エンジニアリング（Platform Engineering）**です。CI/CDやインフラの仕組みを「インフラチームが管理するもの」から「全員がセルフサービスで使える工房」へと昇華させる——これが現代の大規模開発組織における生産性の鍵です。

---

## ゴールデン・パス：最高の舗装道路を敷く

プラットフォーム・エンジニアリングの中心にある考え方が**ゴールデン・パス（Golden Path）**です。「Spotifyが普及させたこの概念は、「正しい方法を最も簡単な方法にする」という発想です。

開発者に「どのフレームワークでも使っていい」「どのデプロイ方法でもいい」と完全な自由を与えると、組織全体でサポートしなければならない組み合わせが爆発的に増えます。逆に「このスタックだけ」と制約しすぎると、特殊なニーズに対応できません。

ゴールデン・パスはその中道です。プラットフォームチームが「これを使えば最高の体験が得られる」という推奨ルートを舗装し、その道を選ぶことが最も楽になるよう整備します。他の道も歩けますが、サポートは限定的になります。

```
# QuestForgeのゴールデン・パス定義例

新しいマイクロサービスを作るなら：
  言語: Python 3.12 (FastAPI)
  コンテナ化: Dockerfile テンプレート自動生成
  CI/CD: GitHub Actions ワークフロー自動設定
  監視: Prometheus + Grafana ダッシュボード自動作成
  シークレット管理: Vault 連携
  
  → `questforge new-service <name>` の1コマンドで完結
```

## IDP（内部開発者プラットフォーム）：工房の実装

ゴールデン・パスを具体化したものが**IDP（Internal Developer Platform：内部開発者プラットフォーム）**です。開発者が必要なインフラをセルフサービスで調達・設定・廃棄できる「自動工房」です。

### Backstageによるポータル

[Backstage](https://backstage.io/)（Spotifyが開発しオープンソース化）は、IDPの中核となる開発者ポータルを構築するフレームワークです。

```yaml
# Backstage スキャフォルダテンプレート
# questforge-service-template.yaml
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: questforge-python-service
  title: QuestForge Pythonマイクロサービス
  description: ゴールデン・パスに沿ったマイクロサービスを即座に生成します
spec:
  parameters:
    - title: サービス設定
      required: [serviceName, ownerTeam, description]
      properties:
        serviceName:
          title: サービス名（例: quest-recommendation）
          type: string
          pattern: '^[a-z][a-z0-9-]*$'
        ownerTeam:
          title: オーナーチーム
          type: string
          enum: [quest-team, hero-team, guild-team, platform-team]
        description:
          title: サービスの目的
          type: string
  steps:
    - id: fetch-template
      action: fetch:template
      input:
        url: ./skeleton
        values:
          serviceName: ${{ parameters.serviceName }}
          ownerTeam: ${{ parameters.ownerTeam }}
    - id: create-repo
      action: publish:github
      input:
        repoUrl: github.com/questforge/${{ parameters.serviceName }}
    - id: register-catalog
      action: catalog:register
      # サービスカタログに自動登録
```

このテンプレートを実行するだけで、GitHubリポジトリの作成・CI/CDワークフローの設定・Kubernetes名前空間の作成・監視ダッシュボードの生成・サービスカタログへの登録が、すべて自動で完了します。「インフラチームへのチケット待ち数日」が「ポータルでの操作3分」になります。

### サービスカタログ：全体を「見える」にする

IDPのもう一つの柱が**サービスカタログ**です。「QuestForgeには今何個のサービスがあり、誰がオーナーで、どんな依存関係があるか」を一覧できます。

```yaml
# catalog-info.yaml（各サービスのルートに配置）
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: quest-recommendation-service
  description: 勇者の履歴から最適なクエストを推薦するサービス
  annotations:
    github.com/project-slug: questforge/quest-recommendation-service
    backstage.io/techdocs-ref: dir:.
    pagerduty.com/service-id: P1234567
spec:
  type: service
  lifecycle: production
  owner: quest-team
  system: questforge-backend
  dependsOn:
    - component:hero-profile-service
    - component:quest-catalog-service
    - resource:questforge-vector-db
```

組織全体のサービスマップが可視化されることで、「このサービスを変更したら何が影響を受けるか」が設計時に把握できます。

## Infrastructure as Code：コードが工房の設計図になる

プラットフォームの信頼性は、設定が**宣言的なコードとしてバージョン管理**されていることで保証されます。

### Terraformによるインフラ定義

```hcl
# QuestForgeのKubernetesクラスタ定義
# infra/main.tf

module "questforge_cluster" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 20.0"

  cluster_name    = "questforge-production"
  cluster_version = "1.30"

  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_subnets

  eks_managed_node_groups = {
    # ゴールデン・パスの標準ノード構成
    standard = {
      instance_types = ["t3.medium"]
      min_size       = 2
      max_size       = 10
      desired_size   = 3
    }
  }

  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
    Owner       = "platform-team"
  }
}
```

Terraformでインフラをコード化することで：

- **変更のレビュー**: `terraform plan` で「何が変わるか」をPRレビューと同じプロセスで確認できる
- **再現性**: 本番と同一の環境をステージングに再現できる
- **監査**: インフラの変更履歴がgitログに残る

## DevEx（開発者体験）を測る：DORAメトリクス

プラットフォームの価値は「ツールの数」ではなく**開発者の体験**で測られます。Googleのエンジニアリング研究グループDevOps Research and Assessment（DORA）が特定した4つのメトリクスが、業界標準の指標として広く使われています。

| メトリクス | 定義 | エリート水準 | QuestForgeの目標 |
|-----------|------|------------|----------------|
| **デプロイ頻度** | 本番へのリリース回数 | 1日に複数回 | 1日1回以上 |
| **変更リードタイム** | コミットから本番稼働まで | 1時間未満 | 1日以内 |
| **変更失敗率** | デプロイがインシデントを引き起こす割合 | 5%未満 | 10%未満 |
| **サービス復旧時間** | インシデントから回復までの時間 | 1時間未満 | 4時間以内 |

これらを計測するには、CI/CDパイプライン（6.2節）とオブザーバビリティ（6.4節）の仕組みが基盤として必要です。

### 開発者アンケート：定性的な体験を定量化する

DORAメトリクスは「速さ」を測りますが、「楽しさ」は測りにくいものです。Googleが提唱する**SPACE フレームワーク**（Satisfaction・Performance・Activity・Communication・Efficiency）を使った定期的な開発者アンケートが、定性的な体験を構造化する助けになります。

```
# 四半期ごとの開発者体験アンケート例

1. ローカル開発環境のセットアップ（1-5段階）
   「新しいメンバーが初日に環境を整えるのにかかる時間は？」

2. デプロイへの自信（1-5段階）
   「変更をmainにマージするとき、どのくらい安心して押せますか？」

3. ツールへの満足度（自由記述）
   「最もフラストレーションを感じるボトルネックは何ですか？」
```

## QuestForgeの実装：段階的なプラットフォーム成熟

プラットフォームは一夜にして完成しません。段階的に育てるアプローチが現実的です。

| フェーズ | 規模 | 取り組み |
|---------|------|---------|
| **Phase 1: 自動化の基盤** | 1〜5チーム | CI/CDの統一・Dockerfileテンプレート・基本監視 |
| **Phase 2: セルフサービス化** | 5〜15チーム | Backstageポータル・Terraformモジュール・サービスカタログ |
| **Phase 3: 内部製品化** | 15チーム以上 | SLO管理・コスト可視化・プラットフォームAPI・DevExチーム設立 |

QuestForgeが5チームに成長した段階でBackstageを導入し、20チームに達した段階でプラットフォームをSLO付きの「内部製品」として運用する——この段階的な育て方が、投資対効果を最大化します。

## AI時代のプラットフォーム・エンジニアリング

AIはプラットフォームそのものも変えつつあります。

**コード生成のガバナンス**: 開発者がAIを使ってコードを生成する量が増えるほど、「生成されたコードがゴールデン・パスに沿っているか」の自動チェックが重要になります。Backstageのテンプレートにセキュリティスキャン・依存関係チェック・コーディング規約チェックを組み込むことで、AI生成コードも同じ品質ゲートをくぐれます。

**インフラのAI支援設計**: 「このサービスの負荷パターンから適切なKubernetesのリソース制限を提案して」とAIに問えば、キャパシティ設計の出発点が得られます。DORAメトリクスのトレンドをAIで分析し、ボトルネックを特定する活用も広がっています。

---

プラットフォーム・エンジニアリングの本質は「開発者が最高の仕事をできる環境を設計すること」です。ゴールデン・パスで「正しい選択を最も楽な選択に」し、IDPで「セルフサービスの自動工房」を実現し、DORAメトリクスで継続的に体験を測る——この三つが揃うとき、チームの規模が10倍になっても開発速度は劣化しません。プラットフォームは、エンジニアが技術に集中するための「魔法の環境」そのものです。
