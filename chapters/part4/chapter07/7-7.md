# 7.7 仮説の具現化——プロトタイピングと検証の技法

## 導入: 作る前に確かめる

「この機能、本当に冒険者に喜ばれるだろうか？」

スクラムのスプリントを走り始める前に、この問いに向き合う時間があります。仮説をそのままコードに落とし込んで数週間後に「違った」と気づくより、**数時間の粗削りな試作（プロトタイプ）**で方向性を確かめる方が、はるかに賢い冒険の始め方です。

プロトタイピングとは、「作り捨てることを前提に、素早く具現化して検証する」という技法です。完成品を目指すのではなく、**「この仮説は正しいか？」** という問いへの答えを最短で手に入れることが目的です。

---

## プロトタイプの4段階

プロトタイプには、目的と精度に応じた段階があります。

### 第1段階: 紙のプロトタイプ（ペーパープロト）

最も速く、最も安い検証手段。紙とペンでUIのスケッチを描き、実際の冒険者に操作してもらいます。

```
QuestForge クエスト作成画面（紙のスケッチ）

┌──────────────────────────┐
│  新しいクエストを作る     │
│  ┌────────────────────┐  │
│  │ クエスト名          │  │
│  └────────────────────┘  │
│  難易度: [★☆☆] [★★☆] [★★★]│
│  期限:  [カレンダー]     │
│  [作成する]              │
└──────────────────────────┘
```

画面の遷移を紙のめくりで表現するだけで、「冒険者が迷いそうな箇所」「そもそもこの操作は必要か？」が浮かび上がります。

### 第2段階: ワイヤーフレーム・クリックスルー

FigmaやMiroを使い、モノクロの低解像度画面を作成します。クリック可能なリンクでページ遷移を再現すれば、本物のアプリのような操作感を試せます。デザインの議論をするより先に、**フロー（流れ）の妥当性**を確かめるのが目的です。

### 第3段階: スパイク（Spike）

技術的な実現可能性が不明な場合、**スパイク**と呼ばれる時間限定の調査コードを書きます。スクラムのスプリントに「技術スパイク」を組み込み、「X技術でYを実現できるか？」を実験します。

```python
# スパイク: リアルタイム通知はWebSocketで実現できるか？
# これは本番コードではなく、検証目的の使い捨てコード

import asyncio
import websockets

async def notify_quest_complete(websocket):
    """クエスト完了通知のスパイク実装"""
    await websocket.send('{"event": "quest_complete", "quest_id": "q-001"}')
    response = await websocket.recv()
    print(f"Response: {response}")  # 応答が来るか確認するだけ

# 結論: WebSocketで双方向通信できることを確認 → 本実装へ進む
```

スパイクの成果物は「コード」ではなく「答え」です。実現できた・できなかった、どのくらいの工数がかかりそうか、という知識こそが価値です。

### 第4段階: PoC（Proof of Concept）

スパイクより少し本格的な「概念実証」です。コア機能だけを最低限動く形で実装し、ステークホルダーや実際の冒険者に見せて反応を確認します。エラー処理もセキュリティも後回し——まず「動く」「価値が伝わる」ことを優先します。

## スクラムとプロトタイピングの連携

### スプリント0：発見の旅

プロジェクト開始直後の「スプリント0」は、開発環境の構築やチームの合意形成に使われます。この期間にプロトタイプを作り、**方向性の仮説を検証する**のが賢明です。

```
スプリント0（2週間）
  Week 1: 要求定義 + ペーパープロト作成
  Week 2: ワイヤーフレーム + 冒険者5人にユーザビリティテスト
  →  スプリント1: 検証済みの仮説をバックログに変換
```

### リファインメントでのスパイク計画

スプリントプランニングの前のバックログリファインメント（6.2節参照）で、技術的な不確実性が高いストーリーを見つけたら、次スプリントに「スパイク」を計画します。

```
バックログアイテム:
  [Story] 冒険者がリアルタイムでパーティの進捗を見られる
  → 不確実性: WebSocketかSSEか？サーバーの負荷は？
  → 計画: 次スプリントに2日間のスパイクを組み込む
```

## AI時代の超高速プロトタイピング

AIはプロトタイピングを劇的に加速します。

### 自然言語からUIへ

「クエスト作成画面を作って」とAIに伝えるだけで、HTML/CSSのプロトタイプが数分で生まれます。これを冒険者に見せ、フィードバックを得て、また修正する——このサイクルを一日に何十回も回せます。

```
【AIへの詠唱】
QuestForgeの「クエスト作成フォーム」のHTMLプロトタイプを作ってください。
フィールド:
- クエスト名（テキスト入力）
- 難易度（★☆☆/★★☆/★★★ の選択）
- 期限（日付ピッカー）
- 報酬経験値（数値）
デザインはシンプルなTailwind CSSを使用し、モバイルフレンドリーに。
```

### 技術的スパイクの自動化

「この技術で実現できるか？」という問いに対しても、AIは参考実装を即座に生成します。人間がスパイクコードをゼロから書く時間を、AIが短縮してくれます。

## 「プロトタイプに夢中になる」罠を避けるには

プロトタイプは使い捨てが原則です。「せっかく作ったから本番に使おう」と思い始めると、テストもエラー処理もないコードが本番環境に紛れ込む結果になります。

| 健全なプロトタイプの特徴 | 問題のある兆候 |
|------------------------|--------------|
| 「これは検証用」と明記されている | コードレビューのプロセスに乗っている |
| 使い捨てブランチ・フォルダに置かれている | mainブランチにマージしようとしている |
| 答えが出たら迷わず削除する | 「ベースにして使いまわせるのでは」と考え始める |

## まとめ

1. **仮説を先に検証する**: 数週間のスプリントを走り始める前に、数時間のプロトタイプで方向性を確かめる。
2. **精度は目的に合わせる**: 紙→ワイヤーフレーム→スパイク→PoCの4段階を、問いの性質に応じて使い分ける。
3. **スクラムに組み込む**: スプリント0の活用と、バックログリファインメントでのスパイク計画が実践の基本。
4. **AIで加速する**: 自然言語からUIやコードを生成するAIで、検証サイクルを劇的に短縮できる。
5. **使い捨ての誓い**: プロトタイプは「答えを得るための道具」であり、本番コードではない。目的を達したら迷わず捨てる。

## AIへの詠唱例

```
QuestForgeに「パーティ（グループ）機能」を追加することを検討しています。
まず以下のプロトタイピング計画を立ててください：

1. ペーパープロトで確認すべき画面フロー（3〜5画面）
2. 技術的不確実性（WebRTC？SignalR？）に対するスパイクの設計
3. PoCとして最低限実装すべき機能リスト

その後、ワイヤーフレーム代わりになるHTMLプロトタイプも生成してください。
```
