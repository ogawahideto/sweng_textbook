## 導入: 自分一人では気づけない「死角」

あなたは自分の書いた文章の誤字脱字に、自分ではなかなか気づけないという経験はありませんか？ コードも同じです。どれほど注意深く書いても、作者の脳内では「意図」が補完されてしまい、客観的な「欠点」が見えなくなります。

かつては、先輩や同僚にレビューを頼むのが一般的でした（今でも重要です）。しかし、いつでも頼めるわけではありません。

そこで、AIを**「24時間待機してくれる専属レビュアー」**として召喚します。彼らは文句も言わず、何度でも、様々な視点からあなたのコードをチェックしてくれます。これは単なるツールではありません。あなたの思考を映し出し、洗練させる「鏡」なのです。

---

## 理論的背景: AIレビューの3つのレベル

AIへのレビュー依頼には、深さに応じた「レベル」があります。適切なレベルを選ぶことが、良いフィードバックを得るコツです。

### レベル1: 校正（Linting & Style）
- **依頼**: 「文法ミスやスタイル違反はない？ 変な変数名はある？」
- **効果**: Linterと同様ですが、より柔軟です。「ここではPythonの慣習的に `get_` よりもプロパティを使うべき」といった、文脈を踏まえた指摘が得られます。

### レベル2: 意図の確認（Safety & Logic）
- **依頼**: 「この関数で、引数がNoneだった場合や、空リストだった場合にエラーにならない？」
- **効果**: バグの可能性、エッジケース（境界値）、セキュリティリスクの発見。「もし攻撃者がここにSQLインジェクションを試みたら？」といったシミュレーションも可能です。

### レベル3: 設計の提案（Design & Architecture）
- **依頼**: 「このロジック、もっとシンプルに書けない？ 将来的にクエストの種類が増えるとしたら、どういう構造にすべき？」
- **効果**: 構造的なリファクタリングの提案。StrategyパターンやTemplate Methodパターンなど、デザインパターンの適用を議論します。ここが最も付加価値の高い領域です。

## 実践: AIとの対話的リファクタリング

QuestForgeの「クエスト報酬計算」ロジックを例に、AIと共にコードを磨き上げてみましょう。

### Step 1: 原石の提示（Before）

以下は、典型的な「成長痛」を抱えたコードです。`if-elif` の連鎖が続き、読むのが辛くなっています。

```python
# 複雑な条件分岐（スパゲッティコード）
def calculate_reward(quest):
    base = 100
    if quest.type == 'HUNTING':
        if quest.target == 'DRAGON':
            return base * 5
        elif quest.target == 'GOBLIN':
            return base * 1.2
    elif quest.type == 'GATHERING':
        if quest.region == 'DANGEROUS':
            return base * 2
        else:
            return base
    # ...延々と続く...
    return base
```

### Step 2: レビューの詠唱

あなたはこう問いかけます。単に「直して」ではなく、**「懸念点」**を伝えている点に注目してください。

> 「このコードは条件分岐が深く、将来クエストの種類が増えた時に保守が大変になりそうです。**Open-Closed Principle（開放閉鎖の原則）**に従って、拡張性を高めるためのリファクタリング案をいくつか提案してください。」

### Step 3: AIの提案（Proposal）

AIは以下のような選択肢（メニュー）を提示してくるでしょう。

1.  **辞書活用**: 分岐ロジックを `Map` に置き換える（簡易的）。
2.  **Strategyパターン**: 計算ロジックを別クラスに切り出す（推奨）。
3.  **ポリモーフィズム**: `Quest` クラス自体を継承して `HuntingQuest` を作る（大規模）。

### Step 4: 対話と決定

あなたは考えます。「継承を使うとDB保存が面倒になりそうだ。辞書だと複雑な計算が書けない。Strategyパターンがバランス良さそうだ」

そして、AIに具体的な指示を出します。

> 「Strategyパターンを採用したコード例を書いてください。`RewardCalculator` インターフェースを定義し、狩猟用と収集用の計算クラスを作ってください。」

### Step 5: 研磨されたコード（After）

```python
from abc import ABC, abstractmethod

# 戦略（Strategy）のインターフェース
class RewardStrategy(ABC):
    @abstractmethod
    def calculate(self, quest) -> int:
        pass

# 狩猟クエスト用の報酬計算戦略
class HuntingRewardStrategy(RewardStrategy):
    def calculate(self, quest) -> int:
        multiplier = 5.0 if quest.target == 'DRAGON' else 1.2
        return int(100 * multiplier)

# 収集クエスト用の報酬計算戦略
class GatheringRewardStrategy(RewardStrategy):
    def calculate(self, quest) -> int:
        return 200 if quest.region == 'DANGEROUS' else 100

# クエスト本体は「計算方法」を持つ（委譲）
class Quest:
    def __init__(self, strategy: RewardStrategy):
        self.strategy = strategy
    
    def get_reward(self):
        # 自分で計算せず、専門家に任せる
        return self.strategy.calculate(self)
```

これで、新しいクエストタイプが増えても、新しい `Strategy` クラスを追加するだけで済みます。既存の `Quest` クラスを修正する必要はありません。これが「美しい」ということです。

## コラム: AIの幻覚と「過剰な美化」

AIリファクタリングには罠もあります。

1.  **ロジックの改変**: 「綺麗にしました」と言いつつ、微妙に計算式を変えてしまうことがあります（例: `>` を `>=` にするなど）。リファクタリング前後で動作が変わらないことを確認する**テスト**が必須です（第5章で詳述）。
2.  **YAGNI違反**: 単純な `if` 文で十分な場所に、壮大なデザインパターンを適用したがることがあります。
    > **あなた**: 「いや、ここはクエスト種類が2つしかないから、まだif文でいいよ。複雑すぎる。」
    
    このように、AIの提案を**却下する**ことも、重要なエンジニアの仕事です。

## まとめ

1.  **AIは壁打ち相手**: 孤独なコーディングに「対話」を持ち込むことで、客観性を確保できる。
2.  **観点を指定する**: 「バグはない？」「読みやすくして」「拡張性は？」と、帽子を被らせるように役割を指定する。
3.  **決定権は人間**: AIの提案の中から、プロジェクトの文脈に最適なものを選ぶのがエンジニアの仕事。

次は、こうしたリファクタリングを一時的なイベントではなく、日々の習慣（ゲーム）にする方法——**「技術的負債の返済ゲーム」**について学びます。
