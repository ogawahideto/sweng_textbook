# 付録A：魔導工房の設計図——この書物を生んだAI共同執筆環境

## はじめに：著者の告白

本書はひとつの実験でもあります。

**この書物自体が、AIエージェントとの協働によって執筆されました。**

Claude Code、Gemini CLI、そして Codex CLI——3つの異なる「ゴーレム」が、本文の草稿生成、コードレビュー、整合性チェック、リソース調査を担いました。人間の著者はそれらを指揮し、判断し、編集しました。

3.5節で語った「AIとの協働」の実践がここにあります。付録として、この執筆環境の構造を公開します。同じ方法で技術書を執筆したい方、あるいはAIエージェントを使った大規模な知的作業に挑みたい方への、ひとつの設計図として。

---

## 全体構成：三柱の魔導師

本プロジェクトでは、3つのAIエージェントを使い分けました。

| エージェント | 役割 | 設定ファイル |
|------------|------|------------|
| **Claude Code** | 主力執筆・コードレビュー・詠唱例の精錬 | `CLAUDE.md` |
| **Gemini CLI** | 草稿の批評・多角的なレビュー | `GEMINI.md` |
| **Codex CLI** | コードスニペットの検証・サンプルコード生成 | （共通環境を参照） |

これら3つは**それぞれ独立したエージェント**ですが、**同じ「真実の書」（共通ファイル群）を参照**します。これにより、エージェントが変わっても、執筆のトーン・進捗・ルールが一貫して保たれます。

## 共通環境：三つのゴーレムを束ねる「召喚の書」

### ディレクトリ構造

```
textbook/
├── .ai/                          # 全エージェント共通の設定・記録
│   ├── agents.md                 # エージェントの役割・能力定義
│   ├── QUICKSTART.md             # 新規エージェント向け早見表
│   ├── workflows/                # 標準ワークフロー
│   │   ├── draft.md              # 草稿生成の手順
│   │   ├── review.md             # レビューの手順
│   │   └── plan-next.md          # 次タスク決定の手順
│   ├── commands/                 # カスタムスキル定義
│   │   ├── commit.md             # /commit スキル
│   │   ├── positive-check.md     # /positive-check スキル
│   │   ├── update-toc.md         # /update-toc スキル
│   │   └── ...                   # その他のスキル
│   ├── context/                  # プロジェクトの背景情報
│   │   ├── project.md            # プロジェクト概要
│   │   └── questforge-spec.md    # サンプルアプリ仕様
│   └── sessions/                 # セッションログ（日次）
│       ├── 2026-01-20.md
│       └── ...
│
├── CLAUDE.md                     # Claude Code 固有の設定
├── GEMINI.md                     # Gemini CLI 固有の設定
│
├── style-guide.md                # 執筆スタイルガイド（全エージェント共通）
├── progress.md                   # 進捗状況（全エージェント共通）
├── plan.md                       # 全体構成・目次案
└── chapters/                     # 執筆原稿
```

### 共通参照ファイルの役割

**`style-guide.md`（執筆の憲法）**

文体・トーン・用語・コードスタイルを定義します。「ポジティブ・ライティング方針」（課題ではなく可能性から語る）もここに記載されています。どのエージェントがどのタイミングで草稿を書いても、同じ「声」になるための要となるファイルです。

**`progress.md`（進捗の水晶球）**

全セクションの状態（⬜未着手 / 🟨AI草稿 / 🟦人間修正中 / 🟩レビュー / ✅完成）をひとつの表で管理します。

エージェントが変わっても「次に何をすべきか」を把握できます。作業完了後は必ずこのファイルを更新するルールにしておくことで、セッションをまたいでも一貫した進捗把握が可能になります。

**`.ai/agents.md`（ゴーレムの仕様書）**

エージェントに期待する振る舞い・出力フォーマット・禁止事項を定義します。草稿生成、レビュー、タスク計画それぞれの応答形式を標準化することで、複数エージェントからの出力の品質を均一化できます。

**`.ai/sessions/`（作業記録の石板）**

セッションのたびに日付付きMarkdownファイルを作成し、以下を記録します：
- 当日の主な活動
- 作成・更新したファイル
- 次のステップ

複数エージェントを使い回す場合、「前のセッションで何をしたか」という文脈の引き継ぎが最大の課題になります。このログがその橋渡しをします。

## エージェント固有の設定

### Claude Code：`CLAUDE.md`

Claude Codeは、プロジェクトのルートに `CLAUDE.md` を置くと、セッション開始時に自動的に読み込みます。このファイルには以下を記載しました：

- プロジェクト概要（タイトル・目的・構成）
- ポジティブ・ライティング方針（NG/OK表現の変換ルール）
- ファイル構成の説明
- セッションログの運用ルール（追記方式）
- **カスタムスキル（スラッシュコマンド）の一覧**
- 自動実行タイミング（セクション追加後に`/update-toc`を走らせる、等）

特に強力なのが**カスタムスキル**です。`.ai/commands/` に置いたMarkdownファイルが、そのまま `/コマンド名` として使えます：

| スキル | 実行内容 |
|-------|---------|
| `/commit` | 変更を論理単位ごとにグループ分けしてコミット |
| `/positive-check` | ネガティブ表現をスキャンして代替案を提案 |
| `/update-toc` | 目次（toc.md）を最新の状態に更新 |
| `/session-log` | 当日のセッションログを更新 |
| `/draw-svg` | 図の草稿をSVG形式で作成 |
| `/estimate-pages` | 原稿の分量からページ数を概算 |

これらのスキルはMarkdownファイルなのでGitで管理でき、複数エージェント間で共有もできます。

### Gemini CLI：`GEMINI.md`

Gemini CLIは、プロジェクトのルートに `GEMINI.md` を置くと設定として読み込みます。Claude Codeとは「同じ共通環境を参照する別のゴーレム」として設計し、GEMINI.mdには以下を記載しました：

- プロジェクト概要とQuestForgeアプリの技術スタック
- 執筆ワークフローの概要（plan→draft→edit→review）
- コミットルール
- Gemini特有の追記メモリ（`Gemini Added Memories` セクション）

Gemini CLIは特に「別の視点からのレビュー」に活用しました。Claude Codeが草稿を書き、Gemini CLIが批評する——人間と二種類のAIが「多角形の目」を持つことで、見落としが減ります。

### Codex CLI

OpenAIのCodex CLIも同じプロジェクト構造を参照できます。Claude CodeやGemini CLIが使う `style-guide.md`・`progress.md`・`.ai/` 配下のファイルは、特定のエージェントに依存しない設計にしています。どのエージェントでも「次に何をすべきか」「どんな文体で書くか」を自律的に判断できます。

## マルチエージェント運用の実践知

### 原則1：共通の「真実の書」を一本に絞る

`progress.md` や `style-guide.md` は**必ず1ファイル**に集約します。エージェントごとに別ファイルを持つと、更新が分散して状態が錯綜します。

### 原則2：エージェントへの指示はファイル参照で行う

```
# 良い詠唱
@style-guide.md に従って、3.1節の草稿を書いてください。

# 避けるべき詠唱
「文体は魔法のメタファーを使って、でも技術的に正確で、
 前後のセクションと繋がるように...」（毎回長い指示を書く）
```

ルールをファイルに外出しすることで、どのエージェントでも同品質の出力が得られます。

### 原則3：セッションログで文脈を引き継ぐ

エージェントはセッションをまたいで記憶を保持しません。`.ai/sessions/` への日次ログが、「前回のゴーレムが何をしたか」を次のゴーレムに伝える唯一の手段です。

```markdown
## セッション1（2026-01-20）
- 3.1節の草稿を生成（3,200字）
- QuestForgeのソートアルゴリズム例を追加
- progress.mdを「🟨 AI草稿」に更新

## 次のステップ
- 人間が3.1節をレビュー・修正
- 3.2節の執筆に着手
```

### 原則4：コミットは論理単位で頻繁に

章の完成を待たず、節ごとにコミットします。エージェントが作業中断・切り替えをしても、Gitの履歴が「どこまで完成したか」の水晶球になります。

## 執筆フロー全体像

```
【タスク決定】
 human → /progress → 次に着手すべき節を確認
      ↓
【草稿生成】
 Claude Code / Gemini CLI → .ai/workflows/draft.md に従って草稿を生成
      ↓
【人間レビュー・編集】
 human → 技術的正確性・体験談・ニュアンスを修正
      ↓
【AI品質チェック】
 Claude Code → /positive-check（ネガティブ表現検出）
             → /chapter-consistency（章内整合性）
      ↓
【進捗更新・コミット】
 Claude Code → progress.md を更新 → /commit でグループコミット
      ↓
【TOC更新・セッションログ記録】
 Claude Code → /update-toc → /session-log
```

## この設計から学べること

本書はソフトウェアエンジニアリングを扱っていますが、この付録自体が**「大規模な知的作業へのAI活用」**の実例です。

重要な洞察をいくつか：

1. **AIは「手を動かす職人」、人間は「設計者と審査員」**
   草稿の生成・検索・整合性チェックはAIが担い、最終判断・方向性・体験談の追加は人間が行う。この分業が機能するほど、執筆のスピードと品質が両立します。

2. **エージェントが変わっても品質が均一になる設計**
   スタイルガイド・ワークフロー・進捗管理が「ファイルに宿る知識」として外部化されていると、どのエージェント（あるいはどの人間のアシスタント）が参加しても同じ品質を維持できます。

3. **3.9節で学んだ「手技（スキル）」はここでも活きる**
   `/commit` `/positive-check` `/update-toc` ——これらのカスタムスキルが、本書の執筆を支えました。繰り返す作業をスキルに定義することで、執筆の「儀式的なノイズ」が消え、創造的な部分に集中できました。
