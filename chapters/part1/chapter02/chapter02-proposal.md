## 構成案C 詳細

### 2.1 どう構造化するか？——設計を描く古代文字（UML後編）

**導入**
- 1.5節で描いた「何をするか」から「どう作るか」へ
- 要求フェーズのUML vs 設計フェーズのUML

**理論的背景**
- クラス図の基本要素（クラス、属性、操作、関連、継承）
- シーケンス図の基本要素（ライフライン、メッセージ、活性化）
- オブジェクト図、状態図（発展的内容として軽く触れる）

**実践例: QuestForgeのドメインモデル**
- Hero（勇者）、Quest（クエスト）、Reward（報酬）、Badge（バッジ）
- クラス間の関係（集約、依存、継承）
- クエスト完了のシーケンス図

**AI時代のアプローチ**
- AIにクラス図を生成させる
- 既存コードからUMLをリバースエンジニアリング

**参考文献**
- マーチン・ファウラー『UMLモデリングのエッセンス』
- クレイグ・ラーマン『実践UML』

---

### 2.2 どう分けるか？——美しい構造の黄金律

**導入**
- 「このコード、なぜ触りやすいんだろう？」
- 美しいコードの共通点を探る

**理論的背景**
- 凝集度（Cohesion）: 関連するものを集める
- 結合度（Coupling）: 依存を最小限に
- SOLID原則
  - S: 単一責任の原則
  - O: 開放閉鎖の原則
  - L: リスコフの置換原則
  - I: インターフェース分離の原則
  - D: 依存性逆転の原則

**実践例: QuestForgeで原則を体験**
- 悪い例→良い例の変換（ポジティブに「さらに効果的な方法」として）
- Questクラスが肥大化した状態からの改善

**AI時代のアプローチ**
- AIにSOLID違反を検出させる
- リファクタリング提案の生成

**参考文献**
- ロバート・C・マーチン『Clean Code』
- ロバート・C・マーチン『アジャイルソフトウェア開発の奥義』

---

### 2.3 どう再利用するか？——デザインパターンの知恵

**導入**
- 「この問題、前にも見たことがある」
- 先人たちが発見した「定石」の世界

**理論的背景**
- デザインパターンとは何か
- GoF（Gang of Four）の23パターン概観
- パターンの3分類（生成・構造・振る舞い）

**厳選パターン解説**（QuestForgeに関連するもの）

| パターン | 分類 | QuestForgeでの用途 |
|---------|------|-------------------|
| Strategy | 振る舞い | 難易度に応じた経験値計算 |
| Observer | 振る舞い | レベルアップ時の通知 |
| Factory Method | 生成 | Quest/Badgeの生成 |
| Repository | 構造 | データ永続化の抽象化 |

**AI時代のアプローチ**
- AIに「このコードに適用できるパターンは？」と聞く
- パターン適用のトレードオフを議論

**参考文献**
- GoF『デザインパターン』
- Head First『デザインパターン』
- マーチン・ファウラー『エンタープライズアプリケーションアーキテクチャパターン』

---

### 2.4 どう守るか？——クリーンアーキテクチャ

**導入**
- 「フレームワークが変わっても、ビジネスロジックは生き残る」
- 変化に強いソフトウェアの秘密

**理論的背景**
- レイヤードアーキテクチャの基本
- オニオンアーキテクチャ / ヘキサゴナルアーキテクチャ
- クリーンアーキテクチャの同心円
- 依存性逆転の実践

**実践例: QuestForgeの4層構造**

```
┌─────────────────────────────────────────┐
│             Presentation                │  ← CLI / Streamlit
├─────────────────────────────────────────┤
│             Application                 │  ← Use Cases
├─────────────────────────────────────────┤
│               Domain                    │  ← Entities, Repositories (Interface)
├─────────────────────────────────────────┤
│            Infrastructure               │  ← Repository実装, 外部サービス
└─────────────────────────────────────────┘
        依存の方向: 外側 → 内側
```

**AI時代のアプローチ**
- AIにアーキテクチャ図を生成させる
- 依存関係の分析と改善提案

**参考文献**
- ロバート・C・マーチン『Clean Architecture』
- アリスター・コックバーン「ヘキサゴナルアーキテクチャ」
