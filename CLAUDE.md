# Claude Code プロジェクト設定

このファイルはClaude Codeがプロジェクトの文脈を理解するためのメモリです。

## マルチエージェント運用

このプロジェクトは **Claude Code** と **Gemini CLI** の2つのコーディングエージェントで執筆を行います。

### 共通リソース（必須参照）

以下のファイルは両エージェント共通の設定です。**タスク開始前に必ず参照**してください。

| ファイル | 用途 | 参照タイミング |
|---------|------|---------------|
| `.ai/agents.md` | エージェントの役割・振る舞い定義 | セッション開始時 |
| `.ai/workflows/draft.md` | 草稿生成の手順 | 新規セクション執筆時 |
| `.ai/workflows/review.md` | レビューの手順 | レビュー実施時 |
| `.ai/workflows/plan-next.md` | 次タスク決定の手順 | タスク完了後 |
| `style-guide.md` | 執筆スタイル | 常時 |
| `progress.md` | 進捗状況 | 常時 |

### タスク別の自動参照ルール

| タスク | 参照すべきファイル |
|--------|-------------------|
| **セッション開始** | `progress.md`, `.ai/agents.md`, 最新の`.ai/sessions/*.md` |
| **草稿生成** | `.ai/workflows/draft.md`, `style-guide.md`, `plan.md`, 対象章の既存ファイル |
| **レビュー** | `.ai/workflows/review.md`, `style-guide.md`, 対象セクション |
| **次タスク決定** | `.ai/workflows/plan-next.md`, `progress.md`, `plan.md` |

### エージェント間の連携ルール

1. **progress.md を常に最新に保つ**: 作業完了後は必ず更新
2. **セッションログで引き継ぐ**: 作業内容を`.ai/sessions/YYYY-MM-DD.md`に記録
3. **コミットを頻繁に**: 論理的な単位ごとにgitコミット
4. **齟齬が生じたら実ファイルを確認**: ログより実際のファイル構造を優先

## プロジェクト概要

- **タイトル**: 『「楽しい」ソフトウェア工学の教科書 - デジタル・アルケミー：伝統の美学とAIで紡ぐ創造の地図』
- **目的**: AI時代のソフトウェア工学を「楽しさ」の視点から学ぶ教科書
- **サンプルプロジェクト**: QuestForge（RPG風タスク管理アプリ）

## 最重要：ポジティブ・ライティング方針

**従来のソフトウェア工学教科書のように課題・問題から入らず、可能性・価値から語る。**

### 変換ルール

| 使わない | 使う |
|---------|------|
| 危険です | 可能性が眠っています |
| 失敗する | 成功に近づく |
| 難しい | 奥深い / 挑戦しがいがある |
| 問題 / 課題 | 機会 / テーマ |
| 罠 / 落とし穴 | コツ / ポイント |
| 悪い例 | シンプルな例 / さらに効果的な方法 |
| 衝突 / 矛盾 | バランス / 両立 |
| 見落とし / 盲点 | 追加できる視点 |

### 見出しの例

- NG: 「なぜ〜は難しいのか」「よくある落とし穴」
- OK: 「〜で広がる可能性」「効果を高めるコツ」

詳細は `style-guide.md` の「ポジティブ・ライティング方針」セクションを参照。

## 執筆ワークフロー

1. `plan.md` で全体構成を確認
2. `progress.md` で次のタスクを確認
3. `style-guide.md` の方針に従って執筆
4. 完成したら `progress.md` を更新
5. gitにコミット

**詳細な手順**: `.ai/workflows/` 内のファイルを参照
- 草稿生成: `.ai/workflows/draft.md`
- レビュー: `.ai/workflows/review.md`
- 次タスク: `.ai/workflows/plan-next.md`

## セッションログ運用（日次更新）

**場所**: `.ai/sessions/YYYY-MM-DD.md`

**更新タイミング**:
- セッション開始時（前回分の確認・補完）
- 作業の区切りごと
- セッション終了時

**記録内容**:
- 概要（1-2行）
- 主な活動（箇条書き）
- 作成・更新したファイル
- 次のステップ

### 重要：追記ルール

**同じ日に複数回更新する場合は、既存の内容を上書きせず追記する。**

```markdown
## セッション1（午前）
- [活動内容]

## セッション2（午後）  ← 追記
- [活動内容]
```

- 各セッションは時間帯や番号で区別
- 前のセッション内容は保持したまま、新しいセクションを追加
- 概要は一日の終わりにまとめて更新してもよい

## カスタムスキル（スラッシュコマンド）

| コマンド | 説明 |
|---------|------|
| `/session-log` | セッションログを更新（日次運用） |
| `/positive-check [パス]` | ネガティブ表現をスキャンして代替案を提案 |
| `/progress` | 進捗状況と次のタスクを表示 |
| `/find-references [節]` | 参考文献を調査して提案 |
| `/update-toc` | 目次（toc.md）を最新状態に更新 |
| `/chapter-consistency [章]` | 章内のセクション間整合性を確認 |
| `/draw-svg [種類] [対象]` | 図の草稿をSVG形式で作成 |

スキル定義: `.ai/commands/`

## 自動実行タイミング

以下のタイミングで該当スキルを実行:

| タイミング | 実行するスキル |
|-----------|---------------|
| セッション開始時 | `/session-log`（前回分の確認・更新） |
| 新しいセクション追加後 | `/update-toc` |
| セクションタイトル変更後 | `/update-toc` |
| 章のレビュー完了後 | `/chapter-consistency`, `/update-toc` |
| セッション終了時 | `/update-toc`, `/session-log`
