# サンプルプロジェクト提案

書籍全体で使用する題材の候補を提案します。

## 題材選定の要件

### 必須要件

1. **全章で活用可能**: 要求工学からデプロイ、チーム開発まで幅広くカバー
2. **段階的成長**: 各章で機能を追加し、複雑化していける
3. **テーマとの親和性**: 「魔法」「錬金術」「アルケミー」と自然に結びつく
4. **初心者フレンドリー**: 技術的な前提知識が少なくても理解できる
5. **ワクワク感**: 作っていて楽しい、使いたくなる
6. **実用性**: 実際に動かせる、使える
7. **AI活用の場面が豊富**: 各章でAI支援の実例を示せる

### 技術的要件

- Python優先（style-guideより）
- Webアプリケーションまたはコマンドラインツール
- データベース、API、テストなど多様な要素を含む
- GitHub/Git での協働開発を示せる

## 提案1: 🎮 クエスト管理システム「QuestForge」

### コンセプト

個人やチームのタスク・プロジェクト管理をRPGゲーム風に楽しくするシステム

### テーマとの親和性

- **魔法の杖**: AIによるタスク自動生成、優先順位付け
- **錬金術**: タスクの組み合わせで成果（経験値）を生成
- **冒険**: プロジェクトをクエストとして表現
- **レベルアップ**: 達成度の可視化、スキルツリー

### 各章での展開例

**第1部（要求工学・設計）**:
- 1.1: ユーザーの「タスク管理が続かない」という願いを分析
- 1.2: クラス図（Quest, Task, User, Reward）
- 1.3: AIとブレスト「どんな報酬が嬉しい？」
- 2.1-2.3: アーキテクチャ設計（MVC、Repository パターン）

**第2部（実装・リファクタリング）**:
- 3.1: AIでCRUD操作の実装
- 3.2: Copilotでバリデーション追加
- 3.3: Python → TypeScript変換
- 4.1-4.3: コードの改善、負債返済

**第3部（テスト・運用）**:
- 5.1: TDDでタスク完了機能
- 5.2: AIでエッジケーステスト生成
- 5.3: 障害注入テスト
- 6.1-6.3: CI/CD、ログ監視、A/Bテスト

**第4部（チーム・倫理）**:
- 7.1: スクラムでチーム開発
- 7.2: プルリクエストレビュー
- 8.1-8.2: ゲーミフィケーションの倫理、継続的改善

### 主要機能（段階的に追加）

1. **基本機能**（第1-2部）:
   - タスク作成・編集・削除
   - カテゴリ分類（クエストタイプ）
   - 期限管理

2. **ゲーム要素**（第2部）:
   - 経験値・レベルシステム
   - 達成バッジ・称号
   - 統計ダッシュボード

3. **AI機能**（全体）:
   - タスク自動生成
   - 優先順位提案
   - 進捗予測

4. **チーム機能**（第4部）:
   - パーティ（チーム）管理
   - クエスト共有
   - リーダーボード

### メリット

✅ 誰もが共感できる（タスク管理は普遍的な課題）
✅ ゲーム要素でワクワク感
✅ 実用性が高い（読者が実際に使える）
✅ AI活用シーンが豊富
✅ 段階的に複雑化しやすい

### デメリット

△ タスク管理ツールは競合が多い
△ ゲーム要素がテーマと少し離れる可能性

## 提案2: 🧪 レシピ管理システム「AlchemyKitchen」

### コンセプト

料理レシピを「錬金術の調合」として管理・共有するシステム

### テーマとの親和性

- **錬金術**: 材料の組み合わせ → 料理の完成
- **魔法の書**: レシピ集 = 魔導書
- **実験**: 新しい組み合わせの試行
- **伝承**: レシピの継承と改良

### 各章での展開例

**第1部**:
- ユーザーインタビュー（どんなレシピ管理が欲しい？）
- ER図（Recipe, Ingredient, User）
- 栄養計算、代替材料提案など

**第2部**:
- レシピ検索機能の実装
- AIで献立提案
- レシピの改良提案

**第3部**:
- 調理手順のバリデーションテスト
- レシピデータベースの負荷テスト
- ユーザー投稿の監視

**第4部**:
- チームでレシピ投稿
- レビュー機能
- オープンソース的なレシピ改善

### 主要機能

1. レシピCRUD
2. 材料管理・在庫チェック
3. 栄養計算
4. AI献立提案
5. 代替材料提案（アレルギー対応）
6. 調理タイマー
7. レシピ共有・評価

### メリット

✅ 「錬金術」との自然な結びつき
✅ 実用性が非常に高い
✅ データモデルが分かりやすい
✅ AI活用（献立提案、代替材料）が自然

### デメリット

△ ドメインが料理に限定される
△ エンジニア以外の読者が多くなる可能性

## 推奨：🎮 クエスト管理システム「QuestForge」

### 推奨理由

1. **最も汎用性が高い**: プロジェクト管理は全エンジニアに関連
2. **ワクワク感**: ゲーム要素で「楽しい」を体現
3. **段階的成長**: シンプル → 複雑への移行が自然
4. **AI活用豊富**: タスク生成、優先順位、進捗予測など多様
5. **実用性**: 読者が実際に使える（書籍のマーケティングにも）
6. **チーム機能**: 第4部のチーム開発に自然に接続

### 次のステップ

1. QuestForgeの詳細仕様を定義
2. 各章での機能展開を詳細化
3. 初期ドメインモデルを作成
4. サンプルコードのディレクトリ構造を決定
