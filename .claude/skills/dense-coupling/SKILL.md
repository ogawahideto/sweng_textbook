# Dense Coupling — 書籍らしい密結合への変換

指定されたMarkdownファイルまたはディレクトリを対象に、3つの変換ルールを適用してセクション間の繋がりを強化します。

## 使い方

```
/dense-coupling [ファイルパス または ディレクトリパス]
```

引数がない場合は現在作業中の章、または `chapters/` ディレクトリ全体を対象にします。

## 変換ルール

### ルール1: `## 導入:` 見出しを削除

```
## 導入: タイトルテキスト

本文の最初の段落...
```
↓
```
本文の最初の段落...
```

見出し行のみ削除。本文・空行はそのまま（余分な空行のみ整理）。

### ルール2: `## 理論的背景` 見出しを処理

- `## 理論的背景: XXX` → `## XXX`（プレフィックスのみ削除）
- `## 理論的背景`（サブタイトルなし）→ 行ごと削除

### ルール3: `## まとめ` の箇条書きを散文に変換

番号付き箇条書き（`1.` `2.`）や `-` 箇条書きを、流れる段落文（2〜3段落）に書き直す。

**変換前:**
```markdown
## まとめ

1. **SOLID原則**は、変更に強い「美しい構造」を作るための指針。
2. **単一責任の原則 (SRP)** を意識するだけで再利用性が向上する。
3. **高凝集・疎結合**な設計は、チームメンバーへの最高のプレゼント。

次は、デザインパターンへ進みましょう。
```

**変換後:**
```markdown
## まとめ

SOLID原則は、変更に強く、拡張するのが楽しくなる「美しい構造」を作るための羅針盤です。
特に単一責任の原則（SRP）を意識するだけで、コードの理解しやすさと再利用性は劇的に向上します。
一つのクラスが一つの仕事に集中する（高凝集）設計は、半年後の自分やチームメンバーへの最高のプレゼントです。

原則を手に入れた次のステップは、それをより具体的なパターンとして活用することです。
2.4節では、先人たちが積み上げてきた知恵の結晶、デザインパターンの世界へと進みます。
```

**末尾には必ず「次節（または次章）への橋渡し段落」を追加する。**
最終節（章の締め）の場合は、次の章への橋渡し段落を追加する。

## 実行手順

### Step 1: 対象ファイルの特定

引数で指定されたパスのMarkdownファイルをリストアップ。
ディレクトリの場合は再帰的に `.md` ファイルを収集。

### Step 2: 変換要否の確認

各ファイルを読み、以下のパターンが存在するか確認:
```bash
grep -n "^## 導入:\|^## 理論的背景\|^## まとめ" [ファイル]
```

変換対象がなければスキップ（ログに「変換不要」と記録）。

### Step 3: 変換を実行

ファイルごとに3つのルールを順番に適用する。

**まとめの散文化の際の注意点:**
- 元の内容のエッセンスはすべて保持する（大幅な削除・加筆をしない）
- スタイルガイド（`style-guide.md`）のポジティブ・ライティング方針に従う
- コードブロック、図の参照 (`![...]`) は変更しない
- `## さらに学ぶためのリソース` セクションは変更しない
- `## AIへの詠唱例` セクションは変更しない

### Step 4: 変更の確認レポート

```markdown
## 密結合変換レポート

| ファイル | ルール1（導入） | ルール2（理論） | ルール3（まとめ） | 状態 |
|---------|--------------|--------------|----------------|------|
| 1-1.md  | ✅ 削除       | ✅ 処理       | ✅ 散文化        | 完了 |
| 1-2.md  | ✅ 削除       | —            | ✅ 散文化        | 完了 |
| 1-3.md  | —            | —            | —              | 変換不要 |

### サマリー
- 処理ファイル数: X
- 変換適用数: Y
- スキップ: Z
```

### Step 5: ユーザー確認後にコミット

変換完了後、ユーザーに確認を求めてから git commit する。
コミットには `/commit` スキルを使用するか、以下の形式で手動コミット:

```
refactor([章番号]): 疎結合→密結合への書き直し
```

## 注意事項

- このスキルは**ブランチ上で実行**することを前提としています
  - 未作成の場合は `git checkout -b editor/dense-coupling` で作成
- プロローグ・エピローグは `## 導入:` がないため通常は変換不要
- 変換後は `/positive-check` で表現の確認も推奨
