5.4 【外伝】万変の魔導書——ソフトウェアプロダクトライン

導入: 一つの成功から、千の多様性へ

リファクタリングの修行を積んだアルケミストは、やがて一つの壁に突き当たります。「目の前のコードは綺麗になった。しかし、似たような別のプロダクトを作るたびに、コードをコピー＆ペーストして少しだけ書き換える作業を繰り返していないか？」という壁です。

私たちが育ててきた QuestForge が大評判となり、「騎士団専用の重厚な管理システム」や「魔法学院向けの学習支援システム」といった、派生プロダクトの依頼が殺到したと想像してください。

それぞれのソースコードをバラバラに管理（クローン）すれば、バグ修正や機能改善のたびに、すべてのコードを渡り歩く地獄が待っています。この難題を解決する高度な錬金術、それがソフトウェアプロダクトライン（SPL）です。

1. 共通の核（コア資産）と変幻する頁

SPLの本質は、システムを「コア資産（Core Assets）」と「可変性（Variability）」に明確に分離することにあります。

  • コア資産（真理の頁）: どのプロダクトでも変わらない、共通のビジネスルールやデータ構造。QuestForgeで言えば「クエストの完了判定」や「経験値の計算ロジック」です。
  • 可変性（選択の印）: プロダクトごとに切り替える機能やUI。例えば「騎士団版のみに存在する上官への報告機能」や「魔法学院版だけの試験評価システム」などです。

リファクタリングを通じて、単一のプロダクトからこれらを抽出していく作業は、あたかも一つの重厚な魔導書を、ページを差し替え可能な「バインダー式の魔導書」へと組み替えるようなものです。

2. 派生から量産へ：リファクタリングの戦略

最初から完璧なプロダクトラインを設計するのは奥深い挑戦です。多くの場合、SPLは以下のリファクタリング・ステップを経て誕生します。

ステップ1: 共通項の括り出し（抽出）

複数の派生プロダクトを比較し、重複しているロジックを共通ライブラリやベースクラスへと吸い上げます。

ステップ2: 変化点のカプセル化（プラグイン化）

プロダクトごとに異なる部分を、第2章で学んだ「Strategyパターン」や「依存性注入（DI）」を使って、外から差し替えられるようにリファクタリングします。

ステップ3: フィーチャーモデルの構築（設計）

「どの機能とどの機能が共存できるか」を整理します。

  • 「騎士団版」なら A機能とB機能は必須、C機能は禁止。
  • 「一般向け」なら A機能のみ。 このように、機能の組み合わせ（レシピ）を定義することで、設定一つでプロダクトを構成（ビルド）できるようにします。

3. プロダクトラインの計測：数字で成果を見る

プロダクトライン化の効果は、いくつかの指標で定量的に把握できます。直感や経験則だけでなく、数字で語れることが、投資判断や設計改善の羅針盤になります。

再利用率（Reuse Rate）

最も直接的な指標は「コア資産をどれだけ使い回せているか」です。

再利用率 = コア資産の行数の合計 / 全プロダクトの全行数の合計 × 100

QuestForgeで3つの派生プロダクトを持つ場合：

プロダクト コア資産（行） 固有コード（行）
騎士団版 3,000 800
魔法学院版 3,000 600
一般向け 3,000 400
合計 9,000 1,800

再利用率 = 9,000 / (9,000 + 1,800) × 100 ≈ 83%

個別開発なら同じ機能を3回書き直すため、総行数は10,800行近くになります。プロダクトラインによりわずか1,800行の差分で3製品を差別化できたことになります。

可変性複雑度（Variability Complexity）

バリエーションポイントが増えるほど、テストすべき組み合わせ数が増加します。

最大組み合わせ数 = 各バリエーションポイントの選択肢数の積
バリエーションポイント 選択肢
報酬システム 3種類（金貨・勲章・単位）
協力プレイ 2種類（なし・あり）
レポート機能 2種類（なし・あり）

最大組み合わせ: 3 × 2 × 2 = 12通り

「12通りすべてのビルドが正しく動作するか」を自動テストで保証できると、プロダクトラインの品質を管理しやすくなります。この数字を「許容できる範囲に収める」ことが、フィーチャーモデル設計の実践的な指針になります。

ROI（投資対効果）の試算

プロダクトライン化には初期コストがかかりますが、製品数が増えるほどメリットが拡大します。

def calculate_spl_roi(
    num_products: int,
    core_dev_effort: float,     # コア資産の開発工数（人月）
    variant_dev_effort: float,  # 各バリエーションの開発工数（人月）
    solo_dev_effort: float,     # 個別開発した場合の1製品あたりの工数（人月）
) -> dict:
    """プロダクトライン化のROIを試算する"""
    spl_total = core_dev_effort + (variant_dev_effort * num_products)
    solo_total = solo_dev_effort * num_products
    savings = solo_total - spl_total
    roi_percent = (savings / core_dev_effort) * 100

    return {
        "spl_total_months": spl_total,
        "solo_total_months": solo_total,
        "savings_months": savings,
        "roi_percent": roi_percent,
    }

# QuestForgeの例
result = calculate_spl_roi(
    num_products=3,
    core_dev_effort=30,    # コア構築に30人月
    variant_dev_effort=5,  # 各バリエーションに5人月
    solo_dev_effort=20,    # 個別開発なら1製品20人月
)
# → spl_total_months: 45  solo_total_months: 60
# → savings_months: 15（25%削減）  roi_percent: 50.0%

製品数が4つ、5つと増えるにつれて、節約効果はさらに加速します。これがプロダクトラインを「戦略的投資」と呼ぶ理由であり、「コアを綺麗に保つ」というリファクタリングの精神が長期的な価値を生み出す仕組みです。

4. アルケミストの報酬：高品質な量産

プロダクトラインを構築する最大のメリットは、「一つの修正が、すべての派生製品に恩恵をもたらす」ことです。

コア資産のバグを直せば、それを利用する騎士団版も、学院版も、一般版も、同時に浄化されます。また、新しい魔法（機能）を追加する際も、コア資産に実装すれば、すべてのプロダクトで即座に利用可能になります。

これは、リファクタリングが「コードを綺麗にする」という目的を超えて、「ビジネスの機動力を極限まで高める」という戦略的な武器に進化した姿なのです。

まとめ：彫刻から金型へ

リファクタリングの初期段階は、一つの石を丁寧に磨き上げる「彫刻」のような作業です。しかし、経験を積んだアルケミストは、その彫刻を「美しい金型（プロダクトライン）」へと昇華させます。

  1. コア資産と可変性の分離: どのプロダクトでも変わらない共通部分と、切り替える部分を明確に分ける。
  2. 漸進的なリファクタリング: 共通項の括り出し → 変化点のカプセル化 → フィーチャーモデルの構築、という3ステップで育てる。
  3. 数字で効果を測る: 再利用率・可変性複雑度・ROIという指標で、プロダクトライン化の成果を定量的に把握する。
  4. 一修正が全製品に波及: コア資産を磨くことで、すべての派生製品が同時に恩恵を受ける構造を作る。

「一つを作って終わり」ではなく、「共通の魂を持つ種族（プロダクトファミリー）」を育てる視点を持つこと。それが、複雑な現代の魔法世界を生き抜くための、高度な設計思想なのです。

コラム：QuestForgeの多世界解釈

もし QuestForge をプロダクトライン化するなら、以下のような「レシピ」が考えられます。

  • ベース: ユーザー管理、クエストDB、進捗計算。
  • バリエーション点:
    • 報酬システム: 「金貨（一般向け）」「名誉勲章（騎士団向け）」「単位（学院向け）」
    • 協力プレイ: 「なし（ソロ版）」「パーティ組成（ギルド版）」

これらをソースコードを汚さずに切り替えられるようになった時、あなたのリファクタリング技術は真のマスターレベルに達したと言えるでしょう。

AIへの詠唱例

IDE統合型：コア資産の抽出分析

@domain/quest.py と @domain/hero.py を読んで、
現在の実装から「どの部分がプロダクトライン化できるコア資産か」を分析してください。
共通化できるロジックと、バリエーションポイントになりうる箇所を具体的に列挙し、
将来の派生プロダクトに向けたディレクトリ構造とインターフェース設計の案を提示してください。

CLIエージェント型：再利用率の計測

sample-code/questforge/ のコードベース全体を走査して、
ソフトウェアプロダクトライン化の観点から以下を分析してください：

1. 再利用率の推定
   - どのモジュール・クラスが「コア資産」候補か（変更頻度が低く、汎用性が高いもの）
   - どの部分が「バリエーションポイント」候補か（プロダクトごとに変わりそうな箇所）

2. 可変性複雑度の評価
   - 現状でif/switchによる分岐が集中している箇所を抽出
   - Strategyパターンや依存性注入で置き換えられる候補を提示

3. 段階的な移行計画
   - 最初に取り組むべき「最もROIが高い共通化」を1つ提案し、
     具体的なリファクタリング手順をコードとともに示してください
